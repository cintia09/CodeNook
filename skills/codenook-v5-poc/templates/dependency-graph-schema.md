# Dependency Graph Schema

Planner writes this file to `.codenook/tasks/<T-xxx>/decomposition/dependency-graph.md`
whenever a parent task has been decomposed into subtasks. The **Queue Runtime
Scheduler** (core §19) reads it to compute the "ready set" — subtasks whose
upstream dependencies are all `done`.

## File format

Plain Markdown. **Required** sections: `## Nodes` and `## Edges`. Other
sections (notes, rationale) may appear but are ignored by the scheduler.

```markdown
# Dependency Graph — <T-xxx>

_Generated by planner at <ISO-8601 timestamp>._

## Nodes

- <subtask_id>: <one-line summary>
- ...

## Edges

- <subtask_id> depends_on <subtask_id>
- ...

## Notes (optional, free-form)

...
```

## Parsing rules (normative)

1. `## Nodes` section: each non-empty line starting with `- ` declares one
   node. Node id is the token **before** the first `:`. Summary (after `:`)
   is informational.
2. `## Edges` section: each non-empty line starting with `- ` declares one
   edge. Line format is exactly:
   ```
   - <child> depends_on <parent>
   ```
   Meaning: `<child>` cannot start until `<parent>` is `done`. Other
   relation words (`needs`, `requires`, `after`) are **not** accepted in v5.0
   POC — keep the vocabulary fixed for the scheduler's `awk`/`grep` parser.
3. No self-loops. No duplicate edges.
4. No cycles. The scheduler detects cycles and refuses to dispatch (raises
   HITL).
5. Max depth: **2 levels** (task → subtask). Sub-subtask decomposition is
   not allowed in v5.0 POC.

## Example

```markdown
# Dependency Graph — T-003

_Generated by planner at 2025-01-20T10:15:00Z._

## Nodes

- T-003.1: Database schema & migrations
- T-003.2: API layer (REST endpoints)
- T-003.3: Business-logic services
- T-003.4: E2E test harness

## Edges

- T-003.2 depends_on T-003.1
- T-003.3 depends_on T-003.1
- T-003.3 depends_on T-003.2
- T-003.4 depends_on T-003.3

## Notes

T-003.1 is the critical path. T-003.2 and T-003.3 share the schema.
```

Ready set after each completion:

| Completed              | Ready (if concurrency allows)     |
|------------------------|-----------------------------------|
| (none)                 | T-003.1                           |
| T-003.1                | T-003.2                           |
| T-003.1, T-003.2       | T-003.3                           |
| T-003.1, T-003.2, T-003.3 | T-003.4                        |
| all                    | (empty → parent integration phase)|

## Relationship to `state.json`

The parent task's `state.json` mirrors subtask status:

```json
{
  "task_id": "T-003",
  "subtasks": [
    {"id": "T-003.1", "status": "done",        "depends_on": []},
    {"id": "T-003.2", "status": "in_progress", "depends_on": ["T-003.1"]},
    {"id": "T-003.3", "status": "pending",     "depends_on": ["T-003.1","T-003.2"]}
  ]
}
```

The graph file is the **normative spec**. `state.json` is a derived cache
refreshed after each tick. If they disagree, graph wins; orchestrator rebuilds
state.
