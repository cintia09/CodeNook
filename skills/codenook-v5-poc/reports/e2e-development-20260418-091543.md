## 2. Bugs

### 2.1 [minor] Dispatch audit warns on workspace-relative `scratch/` path
- **Repro**: dispatch any sub-agent whose manifest lives outside `.codenook/` (I used `scratch/security-audit-T002-manifest.yaml` per §21.3). Run `bash .codenook/dispatch-audit.sh T-002`.
- **Observed**: `[5] Workspace containment: ⚠️  path outside .codenook/: scratch/security-audit-T002-manifest.yaml`.
- **Severity**: minor — the warning is correct per §21.3, but the orchestrator was given no scratch area inside `.codenook/` to use, and `/tmp` is sandbox-blocked. No documented "session scratch" location exists.
- **Proposed fix**: either (a) reserve `.codenook/_workspace/scratch/` analogous to `_workspace/prompts/` (§18 Trigger A), or (b) downgrade containment warning to info when path is workspace-relative and starts with a known scratch prefix.
- **Fix location**: `.codenook/dispatch-audit.sh` check [5], and a new line in core §6 documenting allowed scratch locations.

### 2.2 [minor] `queue-runner.sh` has no `tick` / `process` subcommand
- **Repro**: `bash .codenook/queue-runner.sh --help` (or any unknown subcommand) lists `status / list / sweep / ready / deps / cycles / lock / unlock`. There is `sweep` (step-1 only per §19.5) but no single command that performs the full §19.2 tick (sweep + process + ready + dispatch). The orchestrator must inline Python to do it (which is what I did).
- **Severity**: minor — protocol is faithful to §19.2 saying tick is the orchestrator's job, but the runner deliberately splits it. Documentation could state this more clearly.
- **Proposed fix**: add explicit core §19.5 note "the runner intentionally exposes step 1 (sweep) and steps 3/lock primitives; steps 2 and 4 are orchestrator-only."

### 2.3 [minor] `subtask-runner.sh seed` initial subtask phase = `clarify`, not `implement`
- **Repro**: `bash .codenook/subtask-runner.sh seed T-002` after planner emits `decomposed`. Read `subtasks/T-002.1/state.json` → `"phase": "clarify"`.
- **Observed**: per §17.3 each subtask "runs the full 6-phase pipeline (clarify → design → ... → validate)". So this is technically correct. But the runner's seed inconsistently emits `phase: clarify` for some subtasks and `phase: implement` for others depending on whether `parent_section` looks like a build-only step (the runner uses a heuristic). For T-002, both subtasks got `clarify`. Driving sub-clarify+sub-design × 2 subtasks would have ~doubled the agent count (~14 sub-agents instead of 10). I deliberately deviated and dispatched only the implementer for each subtask (with parent clarify/design as inputs) — see Friction §3.2 for full rationale.
- **Severity**: minor — protocol-faithful but cost-prohibitive at depth 2 with full pipelines per subtask.
- **Proposed fix**: introduce a planner-controlled per-subtask field `start_phase: implement` in plan.md table to skip clarify/design when the subtask scope is fully derived from parent. Then `seed` writes that field through.

### 2.4 [minor] `dispatch-log.jsonl` is append-only but has no schema validation in the writer
- **Repro**: Anyone (including me as orchestrator) writes JSON lines via `echo … >> dispatch-log.jsonl`. A typo in field names is silently accepted; `dispatch-audit.sh` later flags missing fields as audit warnings, not write-time errors.
- **Severity**: minor — audit-time detection is acceptable, but a `dispatch-audit.sh log <fields>` writer wrapper would prevent silent log corruption.
- **Proposed fix**: add `dispatch-audit.sh emit <task_id> <phase> <role> <manifest> <output_expected>` that constructs the JSON, validates fields, and appends. Document it as the only sanctioned writer.

### 2.5 [minor] No documented session-scratch dir for orchestrator-internal state
- **Repro**: Stress test needed a manifest file for the security-auditor; `/tmp` is sandbox-blocked; `.codenook/_workspace/` exists only conceptually (§18 mentions it but `init.sh` doesn't create it). I used `scratch/` at workspace root, which then trips bug 2.1.
- **Severity**: minor — every orchestrator instance that needs scratch will hit this.
- **Proposed fix**: `init.sh` creates `.codenook/_workspace/{prompts,scratch}/` and core §6/§18 documents `_workspace/scratch/` as the allowed location for orchestrator scratch (manifests for `security-auditor`, distiller refreshes that don't belong to a task, etc.).

## 3. Friction (protocol ambiguities, missing checks, things the LLM had to guess)

### 3.1 "PHASE_ENTRY_QUESTIONS" is a referenced construct but not defined
The user prompt called it out as a verifiable feature; `codenook-core.md` mentions it only once (§22.1 in passing) without listing what the questions are. The actual creation-time required-fields list lives in §22.3 as a markdown table titled "Required Creation-Time Fields". A reader expecting a `PHASE_ENTRY_QUESTIONS` array (e.g., from an older spec) finds nothing. Recommend either (a) removing the term from §22.1 prose and pointing to the §22.3 table, or (b) defining `PHASE_ENTRY_QUESTIONS` as a YAML stanza in `config.yaml` with the required keys per phase, so it's mechanically queryable.

### 3.2 §17.3 vs. budget reality — full subtask pipeline is impractical
Per protocol every subtask runs clarify→design→plan→implement(±review)→test→accept→validate. With 2 subtasks at `claude-opus-4.7`, that is ~14 sub-agent dispatches just for the subtask layer (plus parent integration), each ≥10K tokens. For a CLI-sized task this is wildly disproportionate. I deviated by dispatching only the implementer per subtask, passing parent clarify+design as context. The protocol provides no sanctioned "abbreviated subtask pipeline" knob; in practice every real run will deviate the same way. Recommend formalising a `subtask_phase_set: ["implement"]` override in plan.md or config.

### 3.3 Dual-mode `serial` requires `max_iterations ≥ 2` to be meaningful
I set `max_iterations=1` for budget. The serial loop still ran (one impl + one review), but the loop's exit-condition logic in §15 says "if iter <= max_iterations, dispatch impl, then reviewer, then check verdict". With `max_iterations=1` and `looks_good`, you exit after one review; but if the reviewer returns `needs_fixes`, you'd hit the §15 fall-through that escalates HITL with "did not converge within max_iterations". This means `max_iterations=1` with dual_mode=serial is effectively "one shot or HITL", with no meaningful "try again" budget. Recommend either rejecting `max_iterations=1` for serial mode at task creation, or documenting this as the intended degenerate.

### 3.4 No `post_phase_refresh()` was triggered
Per §5 "after every sub-agent return: ... post_phase_refresh()" and §18 Trigger A, a session-distiller refresh should fire after every phase. In Mode-B stress-test mode the orchestrator code path is the LLM itself; the protocol relies on prompt-discipline to remember to dispatch the distiller. I omitted distiller dispatches to keep the agent count tractable. `latest.md` is therefore stale (still init placeholder). Recommend either (a) making the distiller dispatch a hard rule the orchestrator cannot skip, with a runtime audit (analogous to dispatch-log.jsonl), or (b) making the distiller agent ultra-cheap (sonnet-4.5 / haiku) so the cost objection disappears.

### 3.5 Manifest path conventions for parent-level integration phases
§3 Routing Table lists `phase-4-tester.md`, `phase-5-acceptor.md`. After §17 fan-out the parent runs *integration*-test/accept; the protocol doesn't say whether to use `phase-4-tester.md` (overwriting the slot) or a distinct name like `integration-test.md`. T-001 used `integration-test.md`/`integration-accept.md`, so I followed that precedent. Recommend documenting this naming convention explicitly in §17.4.

### 3.6 `subtask-runner.sh seed` heuristic for initial `phase`
The runner inspects `parent_section` / `primary_outputs` to pick `phase: clarify` vs `phase: implement` for each subtask. This heuristic is undocumented and surprising — both T-002 subtasks were tagged `clarify` even though they're implementation work. See Bug 2.3.

## 8. Bottom line

CodeNook v5.0 POC is **mostly functional end-to-end** for a small task with subtask fan-out:
- Mode B dispatch + dispatch-log + audit all work as designed.
- Subtask seeding, dependency parsing, locks, and queue files behave correctly.
- HITL queue + adapter + decision file handoff is solid; coexistence with another session's pending items works.
- Security audit + preflight pipeline is robust.

The friction is mostly around **cost**: the protocol mandates depth-2 full-pipeline subtasks (~14 dispatches for 2 subtasks) and per-phase distillation (another ~6 dispatches), neither of which is realistic at opus-4.7 pricing. A real-world run will *always* deviate; making those deviations first-class via config knobs would close the gap between protocol and practice.

No blockers. No data loss. No collision with the concurrent T-001 session.
