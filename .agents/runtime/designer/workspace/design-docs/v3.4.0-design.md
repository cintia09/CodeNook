## T-039: Unify 3-Phase into Agent Workflow

### Current State
- Two workflow modes: "simple" (10 states) and "3-phase" (18 states)
- `agent-init` asks user to choose between them
- Different FSM transition tables for each mode
- `agent-orchestrator` only drives 3-phase mode

### Design Decision: Merge into ONE Unified Flow

**ADR**: We choose to merge both modes into a single workflow that uses the simple linear flow as the backbone, but incorporates key 3-phase elements:

| 3-Phase Element | Keep/Drop | Rationale |
|----------------|-----------|-----------|
| Phase 1 design_review gate | **Keep** → add optional reviewer step after designing | Important quality gate |
| Phase 2 parallel tracks | **Drop** | Too complex for most tasks; can be manually enabled |
| Phase 2 convergence gate | **Drop** | Only needed with parallel tracks |
| Phase 3 staged testing (regression→feature→log) | **Drop** | Over-engineered; tester handles this internally |
| Feedback loops with counter | **Keep** | Useful safety mechanism |
| DFMEA output | **Keep** → move to T-040 | Valuable for quality |

**Unified FSM** (11 states):
```
created → designing → implementing → reviewing → testing → accepting → accepted
              ↑            ↑                ↓            ↓         ↓
              │            └── fixing ←─────┘            │     accept_fail
              └────────────────────────────────────────────┘
              
ANY → blocked (manual unblock)
```

Changes from current simple mode:
- Add `feedback_loops` counter + `feedback_history` array to ALL tasks (not just 3-phase)
- Auto-block at `feedback_loops >= 10`
- Remove `workflow_mode` field from task schema
- Remove `phase`, `step`, `parallel_tracks` fields from task schema

**Files to modify**:
- `skills/agent-init/SKILL.md` — Remove workflow mode choice (Step 1d)
- `skills/agent-fsm/SKILL.md` — Remove 3-phase transition table; keep unified table + feedback loop
- `skills/agent-orchestrator/SKILL.md` — Simplify to drive unified flow only
- `agents/*.agent.md` — Remove 3-phase state references from skills list

**Migration**: Existing 3-phase tasks keep their status; FSM maps legacy states to unified equivalents:
- `requirements` → `designing`
- `architecture` → `designing`
- `tdd_design` → `designing`
- `dfmea` → `designing`
- `design_review` → `reviewing`
- `test_scripting` → `implementing`
- `code_reviewing` → `reviewing`
- `ci_monitoring` → `testing`
- `ci_fixing` → `fixing`
- `device_baseline` → `testing`
- `deploying` → `implementing`
- `regression_testing` → `testing`
- `feature_testing` → `testing`
- `log_analysis` → `testing`

## T-040: DFMEA Template + Implementer Integration

### Current State
- 3-phase has a `dfmea` state but no actual template
- Implementer has no DFMEA output requirement in simple mode

### Design

**DFMEA Template** (`.agents/templates/dfmea-template.md`):
```markdown
# DFMEA — T-NNN: [Task Title]

| # | Component | Potential Failure Mode | Effect | S | O | D | RPN | Recommended Action | Status |
|---|-----------|----------------------|--------|---|---|---|-----|-------------------|--------|
| 1 | | | | | | | | | pending |

**Legend:**
- S (Severity): 1-10 (10 = catastrophic)
- O (Occurrence): 1-10 (10 = very frequent)
- D (Detection): 1-10 (10 = undetectable)
- RPN = S × O × D (>100 = high risk, must address)
- Status: pending | mitigated | accepted_risk
```

**Integration with implementer**:
- Before FSM transition `implementing → reviewing`:
  - Check: `.agents/runtime/implementer/workspace/T-NNN-dfmea.md` exists
  - If missing: warn (mode=warn) or block (mode=strict) based on project config
- Implementer skill updated to include DFMEA as mandatory output
- Reviewer skill updated to check DFMEA RPN scores during review

**Files to modify**:
- Create `.agents/templates/dfmea-template.md`
- `skills/agent-implementer/SKILL.md` — Add DFMEA output step
- `skills/agent-fsm/SKILL.md` — Add DFMEA document gate
- `skills/agent-reviewer/SKILL.md` — Add DFMEA review checklist item

## T-041: Human-in-the-Loop Review Gate (Core)

### Current State
- FSM transitions are agent-driven (no human approval required)
- Acceptor does manual verification at `accepting` stage only
- No intermediate human checkpoints

### Design: HITL Gate Architecture

**Core Concept**: Each agent, before transitioning to the next state, must:
1. Publish its output document to a review surface
2. Wait for human approval
3. Collect feedback if not approved → modify → re-publish
4. Only transition when approved

**New Skill**: `agent-hitl-gate` (Human-in-the-Loop Gate)

**Gate Configuration** (per-project, stored in `.agents/config.json`):
```json
{
  "hitl": {
    "platform": "local-html",
    "gates": {
      "acceptor": { "enabled": true, "output": "requirements" },
      "designer": { "enabled": true, "output": "design-doc" },
      "implementer": { "enabled": true, "output": "code-summary + dfmea" },
      "reviewer": { "enabled": true, "output": "review-report" },
      "tester": { "enabled": true, "output": "test-report" }
    },
    "auto_approve_timeout_hours": null
  }
}
```

**Gate Flow**:
```
Agent completes work
  ↓
agent-hitl-gate publish(task_id, output_doc)
  ↓
Platform adapter creates review page
  ↓
Agent prints: "📄 Review page ready: <URL>. Waiting for approval..."
  ↓
Agent polls or waits for signal:
  - Local HTML: watch file for approval marker
  - Confluence: poll page comments for "approved"
  - GitHub: poll Issue/PR for approve reaction or comment
  ↓
If feedback received (not approved):
  → Agent reads feedback
  → Agent modifies document
  → Re-publish (loop)
  ↓
If approved:
  → Agent proceeds with FSM transition
  → Gate records approval in task metadata
```

**Task Board Integration**:
```json
{
  "hitl_status": {
    "current_gate": "designer",
    "review_url": "file:///path/to/.agents/reviews/T-038-designer.html",
    "status": "pending_review",
    "feedback_rounds": 1,
    "approved_at": null,
    "approved_by": null
  }
}
```

**Files to create/modify**:
- Create `skills/agent-hitl-gate/SKILL.md` — New skill
- Create `.agents/templates/review-page.html` — HTML template
- Modify all 5 agent skills — Add HITL gate call before FSM transition
- Modify `skills/agent-init/SKILL.md` — Ask user about HITL platform during init
- Modify `skills/agent-fsm/SKILL.md` — Add HITL approval guard

## T-042: Interactive Page Adapters

### Design: Pluggable Adapter Interface

**Adapter Contract** (each adapter implements):
```
publish(task_id, agent_role, content_md) → review_url
poll_status(task_id, agent_role) → { status, comments[] }
get_feedback(task_id, agent_role) → { approved: bool, comments: string[] }
```

### Adapter 1: Local HTML (Default)

**How it works**:
1. Generate HTML from markdown content using template
2. Save to `.agents/reviews/T-NNN-<role>.html`
3. HTML includes:
   - Rendered document content
   - Comment textarea per section
   - "Request Changes" button → writes feedback to `.agents/reviews/T-NNN-<role>-feedback.json`
   - "Approve" button → writes `{"approved": true, "at": "ISO8601"}` to feedback file
4. Agent opens with `open` (macOS) / `xdg-open` (Linux)
5. Agent polls feedback JSON file for changes

**Implementation**: Pure HTML + vanilla JS (no dependencies). Self-contained single file.

### Adapter 2: Confluence

**How it works**:
1. Convert markdown to Confluence wiki markup (or HTML)
2. POST to Confluence REST API: `POST /rest/api/content`
3. Poll comments: `GET /rest/api/content/{id}/child/comment`
4. Check for approval keyword in comments ("approved", "LGTM", "✅")

**Config required** (`.agents/config.json`):
```json
{
  "hitl": {
    "platform": "confluence",
    "confluence": {
      "base_url": "https://company.atlassian.net/wiki",
      "space_key": "PROJ",
      "parent_page_id": "12345",
      "auth": "env:CONFLUENCE_TOKEN"
    }
  }
}
```

### Adapter 3: GitHub Issues

**How it works**:
1. Create GitHub Issue with markdown content: `gh issue create --title "Review: T-NNN <role>" --body "<content>"`
2. Add label: `hitl-review`
3. Poll for comments: `gh issue view <number> --comments`
4. Approved when: comment contains "approved"/"LGTM" or 👍 reaction added

**Files to create**:
- `scripts/hitl-adapters/local-html.sh`
- `scripts/hitl-adapters/confluence.sh`
- `scripts/hitl-adapters/github-issue.sh`
- `.agents/templates/review-page.html`

## T-044: Role Mismatch Detection + Switch Prompt

### Current State
- Agents have defined scope but nothing stops user from asking wrong things
- User must manually realize they're in the wrong role

### Design

**Change**: Add a "scope check" section to each agent skill that lists out-of-scope patterns and maps them to the correct agent.

**Implementation in each skill** (new section: "## Role Mismatch Detection"):

```markdown
## Role Mismatch Detection

When the following intents are detected, prompt the user to switch roles:

| User Intent Pattern | Recommended Role | Detection Keywords |
|---------------------|-----------------|-------------------|
| Write/modify code | implementer | "implement", "write code", "modify", "fix", "code" |
| Review code | reviewer | "review", "check code", "inspect" |
| Run tests | tester | "test", "verify", "validate" |
| Collect requirements | acceptor | "requirement", "new feature", "request" |
| Design architecture | designer | "design", "architecture", "architect" |

When detected:
1. Display: "⚠️ This task is better suited for <recommended role>. Current role: <current role>"
2. Ask: "Switch to <recommended role>?"
3. If confirmed → execute agent-switch
4. If declined → continue with current role (not enforced)
```

**Files to modify**:
- All 5 agent skills: Add "Role Mismatch Detection" section with role-specific patterns
